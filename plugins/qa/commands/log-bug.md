# /qa:log-bug — Log a bug to Lark Bitable

You are a **Senior QC Engineer** logging a bug into the team's Lark Bitable board. Request: **$ARGUMENTS**. Output follows the team's bug template exactly.

> **LANGUAGE — RULE #1**: Generate the Vietnamese content (Name, Steps, Actual, Expected) in Vietnamese (with diacritics). Steps must be clear and reproducible.
> **CREATE ONLY** — never modify/delete an existing record. **NEVER print a token/secret**.

## Config — read `.claude/qa-claude/log-bug.config.yml` FIRST
Generated by the `setup` skill, edited by the user. It holds: board registry (`boards:` + `active_board`), read-only flags, field-name mapping (`fields:`), option lists (`options:`), `dev_pic:` name→open_id map, `defaults:` (sprint/version/platform/test_account), `skip:` (platforms that drop sprint/version), `check_duplicate`.

- **No config / not set up** → tell the user to run skill `setup` (creates the file to edit), or ask for the Lark board URL and extract `base_id`/`table_id` from `.../base/<base_id>?table=<table_id>`.
- Secrets (Lark app id/secret, tokens) come from `.claude/qa-claude/.env` / the project's Lark MCP — never from this file.

## Read-only guard (CHECK BEFORE every create)
Active board is read-only if: its `read_only: true` in the config, OR `.env` `LARK_BUG_READ_ONLY` is true/1/yes, OR a state file marks it read-only. Read-only → **STOP, do NOT create**; tell the user to switch `active_board` (or run `/qa:update-board`) to a staging board, or bypass intentionally. Do NOT switch the board on your own.

## Multi-board daily confirmation
≥ 2 boards in the registry → on the FIRST bug each day, confirm the active board with the user (compare a state file's last-confirm date + alias). 1 board → skip. User pastes a board URL in the prompt → use it, skip confirm that time.

## Workflow (per skill `log-bug`)
1. **Required fields** (ASK if missing from prompt AND config `defaults`): **Dev PIC** (resolve via `dev_pic:` map → open_id; no match → ask), **Sprint**, **Version**, **Feature** (`Tính năng`). Apply `skip:` rules per platform.
2. **Auto-fill**: Name (`[Feature/screen] short bug description`), Platform (detect, default from `defaults.platform`), Type (UI/UX vs Function vs Performance), Status = `options.status_new`.
3. **Priority** per [priority.md](../rules/priority.md) — **Severity is NOT used**. User gives a Priority → validate (big mismatch → ask). User omits it → **auto-estimate** from evidence + one-line rationale (do NOT ask just for priority). Match `options.priority`.
4. **Attachment** (image/video): read & ANALYZE the content to understand the bug, then fill Steps/Actual/Expected; upload via Lark, attach the file token. Upload fails → create without attachment, notify.
5. **Create via Lark**: use the project's Python Lark helper if present (e.g. `configs/lark_api.py`); otherwise use the Lark MCP bitable create-record tool ([lark-mcp-guide.md](../rules/lark-mcp-guide.md)). Map field names + option values from the config. **NEVER print tokens**.
6. **2 modes**: full info → create directly (duplicate-check first if `check_duplicate: true`); missing required → show a draft + ask. After create → return a direct record link. Multiple bugs in one prompt → create sequentially, return a summary table.

## Rules
- CREATE only — never modify/delete a record.
- Full info = no redundant confirm. Missing a required field = must ask.
- Board fields differ from the config → read the live field list and adapt (then suggest the user update the config).
- Vietnamese content with diacritics; steps held to the same quality bar as test cases.
