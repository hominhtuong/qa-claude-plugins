# Skill: exploratory-method

Reusable capability: probe a feature → **find app bugs** → bug report for the dev + gate decision. Platform-agnostic: the command supplies the driver (web → `navigate-web` skill, app → `navigate-app` skill) and the element extractor (`find-elements-web`/`-android`/`-ios`). Standards: [failure-triage.md](../../rules/failure-triage.md), [exploratory-bug-report-template.md](../../rules/exploratory-bug-report-template.md).

> ⚠️ **The app may be broken** (a port/incomplete build) → do NOT assume the app is correct by default. Every wrong observation **MUST** be triaged with a label, not skipped, not guessed.

> 🎯 **Spec is the oracle (when available)**: if the command read a spec first (`results/<feature-name>/analysis.md` — built from local file / URL / Figma / pasted prompt via the `lark-reader` / `figma-reader` / `spec-reader` agents), **probe against it**: each functional requirement, business rule, validation rule, and UI state becomes a concrete check of *actual vs expected*. A confirmed deviation = `[APP-BUG] spec-mismatch`; a behavior the spec doesn't cover = `[NEEDS-TRIAGE]` + a question for PO. With no spec, fall back to the heuristic probes below (still valid, lower bug-catch power) — never block on a missing spec.

## Procedure
0. **Load the oracle (if present)**: read `results/<feature-name>/analysis.md` and turn it into a probe checklist (requirements → flows to run, business rules → figures to verify, validation → valid/invalid/boundary inputs + expected messages, UI states → states to confirm, edge cases → cases to hit). This checklist directs steps 3–6. **Only start probing once the spec is read and the checklist is clear** (the command does the read+analysis before handing off here).
1. **Stand on the right feature screen**: use the driver skill the command supplies (`navigate-app`/`navigate-web`) to open the app, go to Home, and navigate to the feature screen — read `sitemap/sitemap.json` first if it exists (it may already know the path).
2. **Record the navigation map** (skill `update-sitemap`) — **for EVERY screen you pass through** (Home → … → feature, and any sub-screens you open): write/update `sitemap/screens/<id>.json` with the `reach` steps (web: route + clicks · app: the tap actions), `parentId` (where you came from), `keyElement`, `route` (web) / `null` (app), and `notes`. **Do this even if the feature is buggy** — the navigation map is the value later cases use to reach the feature. Regenerate `sitemap.json` at the end (`scripts/gen_sitemap.py`).
3. **Probe the main interactions** (oracle-first when a checklist exists, else heuristic): walk each spec requirement/business-rule/validation/UI-state and compare actual vs expected; plus the always-on heuristics — open/close, filter, create/edit, **open an item's detail**, validate **empty/boundary** input (negative values, max length, odd characters). Note what's correct/wrong, crashes, UI bugs, edge cases.
4. **Read figures/messages CRITICALLY**: wrong totals, nonsensical negative numbers, English text leaking through, raw SQL/exception/stacktrace exposed in the UI, context-wrong messages — **check computed figures against the spec's formulas/business rules** when the oracle gives them. Extract **verbatim** (via page source) to paste into the report.
5. **Re-confirm before you call it a bug (anti-flake)** — most "bugs" on web/app are really **timing**: an API still in flight, data not refreshed yet, an element not mounted. **Never capture-and-report on the first wrong observation.** Before labeling anything `[APP-BUG]`:
   - **Wait, then re-check**: give it ~1s (web: `browser_wait_for` on the expected text/state or a network-idle, not a blind sleep; app: poll the element) and re-read. If the correct state appears after the wait → it was a load delay, **NOT a bug** (note it as a slow-load observation if it's egregiously slow, else drop it).
   - **Reproduce ≥2×**: redo the exact steps from a clean state. A symptom that won't reproduce is **not yet a bug** — keep it as `[NEEDS-TRIAGE]` with the steps you tried, don't put it in the bug section.
   - **Vary the path**: try at least one other route to the same state (reload the page/screen, re-open detail, different entry point, fresh data row). A real bug survives the variations; a one-off timing glitch doesn't.
   - Record the **exact reproduction steps + how many times it reproduced** — that's what makes the report actionable and stops "I saw it once but can't show it again".
6. **On-the-spot triage** ([failure-triage.md](../../rules/failure-triage.md)): only after step 5 confirms it's reproducible → decide **`[APP-BUG]`** (app is wrong vs spec/expectation — *reproducible after the wait+retry*; tag *spec-mismatch* when it contradicts the oracle) vs `[FRAMEWORK]` (we grabbed the wrong element/automation) vs `[ENV]`/`[DATA]`. A symptom that vanished after the wait/retry is a timing artifact, not `[APP-BUG]`. Behavior the spec doesn't cover → `[NEEDS-TRIAGE]` + question for PO. Only `[APP-BUG]` goes into the bug section.
7. **Capture evidence** (skill [capture-screenshot](../capture-screenshot/SKILL.md), rule [screenshot-evidence.md](../../rules/screenshot-evidence.md)) — this run is screenshot-heavy, so **respect the image budget from the start or you will lose the ability to view ANY screenshot mid-run**:
   - **Save full-res evidence to a FILE** named by **BUG-ID** under `results/<feature-name>/screenshots/<BUG-ID>.png` (e.g. `02-APP01-published-tab-crash.png`) — app: `adb -s <serial> exec-out screencap -p > <path>` / `xcrun simctl io <udid> screenshot <path>`; web: `browser_take_screenshot` with the **full relative path** as `filename` (a bare name lands in the MCP output dir). Each `[APP-BUG]` ≥1 image; also a few `..-ok.png` proofs of "checked".
   - **To VIEW**, capture small (app: `appium_screenshot` with `maxWidth: 720`; web: prefer `browser_snapshot`) and **downscale any file before Reading it** (`scripts/downscale_image.py <path> --json` → Read the `.small.png`). View sparingly — use page source / accessibility-id for "which screen / did the value change".
   - **If the many-image cap trips** ("image dimensions exceed max allowed size…"), stop Reading images and **finish the sweep blind** via `appium_get_page_source` / `appium_find_element` (accessibility-id = on-screen label) / `browser_snapshot`, still saving full-res files for the report.
8. **Bug report for the dev**: write `results/<feature-name>/dev-bug-report-<ddMMMyyyy>.md` in the **exact format** of [exploratory-bug-report-template.md](../../rules/exploratory-bug-report-template.md) — each bug: Screen · Symptom (verbatim) · Root cause (if found) · Impact · Expectation · Evidence · Defect ID. Include a **✅ Checked — NO bug** section + **❓ NEEDS-TRIAGE** + environment notes. Write the report in the **configured output language** (`.plugin.env` `LANGUAGE`, **default Vietnamese with diacritics**) — see [output-language.md](../../rules/output-language.md); never translate a verbatim quote.
9. **Sync the register**: append each `[APP-BUG]` into the cross-feature register `results/bug-summary.md` (assign APP-ID, update the per-function count table + severity distribution).

## Depth & coverage for planning (go deep, not one lap)
A shallow "walk around once" sweep produces an incomplete plan with missing cases — the failure to avoid. When the exploratory feeds `/qa:plan-tests`, probe in **depth across contexts**, not breadth-only:
1. **Cross-cutting / app-wide settings that change a feature's behavior** — find and account for the common toggles/config that ripple through the whole app, because each one multiplies the cases the plan must cover. Examples (domain-specific, generalize to the app under test): **tax** — multiple tax types, each affecting both product and sales flows; **e-invoice settings** on/off; the **"a product must have a unit before it can be sold"** rule. For each such setting, note which flows change and what the expected behavior is under each state.
2. **Every variant/type of an entity, with its own actions** — an entity often has several kinds, each with different available actions. Don't probe one kind and generalize. Example: a **product has 3 kinds — normal · attributes/variants · conversion**; opening *detail* and **Copy** must be exercised for **all three** (copy-normal, copy-variant, copy-conversion, …), since each behaves differently. Enumerate the kinds first, then run the key actions (open detail, create, edit, copy, delete) **per kind**.
3. **Combinations that matter** — where a cross-cutting setting (1) meets an entity variant (2), probe the meaningful combination (e.g. a conversion product while e-invoice is on). Don't aim for a full cartesian explosion; cover the combinations the spec/business rules say are distinct.
Record what each context/variant requires so the plan can list a case per real scenario, not a single representative case.

> 🏁 **Run the FULL sweep before deciding.** Steps 3–7 cover the *whole* feature: a wrong observation mid-way is **re-confirmed (wait+retry+vary), and only if it survives → logged + triaged + screenshotted, then you keep going** — never abort the exploration on the first (or any) bug. Only once every probe in the checklist is done do you write the report (8–9) and take the GATE below. The goal is a *complete* picture of the feature, not the first defect.

## GATE decision (only after the full sweep)
- 🔴 **Has `[APP-BUG]`** → the feature is NOT done. **Bug report for the dev is always a deliverable** (all bugs found in the sweep). Whether automation also proceeds is decided by the **caller**, not here: `/qa:plan-tests` defaults to *plan-anyway* (declarable+actionable elements → `KNOWN-BUG` cases asserting the canonical expectation; the rest → Blocked cases) and only stops at the bug report under `--stop-on-bug`. When run standalone, just report the bugs and surface this choice.
- 🟢 **No `[APP-BUG]`** (only possibly FW/ENV/DATA) → app is correct → **extract clean elements + declare the Screen** → ready for **`/qa:plan-tests`**.

> *Automation runs only when the app is already correct.* Planning is the `plan-method` skill; automation errors (FW) are fixed via the `fix-by-layer` skill, NOT recorded in the app bug section.
