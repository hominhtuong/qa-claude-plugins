# Skill: exploratory-method

Reusable capability: probe a feature → **find app bugs** → bug report for the dev + gate decision. Platform-agnostic: the command supplies the driver (web → `navigate-web` skill, app → `navigate-app` skill) and the element extractor (`find-elements-web`/`-android`/`-ios`). Standards: [failure-triage.md](../../rules/failure-triage.md), [exploratory-bug-report-template.md](../../rules/exploratory-bug-report-template.md).

> ⚠️ **The app may be broken** (a port/incomplete build) → do NOT assume the app is correct by default. Every wrong observation **MUST** be triaged with a label, not skipped, not guessed.

> 🎯 **Spec is the oracle (when available)**: if the command read a spec first (`results/<feature-name>/analysis.md` — built from local file / URL / Figma / pasted prompt via the `lark-reader` / `figma-reader` / `spec-reader` agents), **probe against it**: each functional requirement, business rule, validation rule, and UI state becomes a concrete check of *actual vs expected*. A confirmed deviation = `[APP-BUG] spec-mismatch`; a behavior the spec doesn't cover = `[NEEDS-TRIAGE]` + a question for PO. With no spec, fall back to the heuristic probes below (still valid, lower bug-catch power) — never block on a missing spec.

## Procedure
0. **Load the oracle (if present)**: read `results/<feature-name>/analysis.md` and turn it into a probe checklist (requirements → flows to run, business rules → figures to verify, validation → valid/invalid/boundary inputs + expected messages, UI states → states to confirm, edge cases → cases to hit). This checklist directs steps 3–5. **Only start probing once the spec is read and the checklist is clear** (the command does the read+analysis before handing off here).
1. **Stand on the right feature screen**: use the driver skill the command supplies (`navigate-app`/`navigate-web`) to open the app, go to Home, and navigate to the feature screen — read `sitemap/sitemap.json` first if it exists (it may already know the path).
2. **Record the navigation map** (skill `update-sitemap`) — **for EVERY screen you pass through** (Home → … → feature, and any sub-screens you open): write/update `sitemap/screens/<id>.json` with the `reach` steps (web: route + clicks · app: the tap actions), `parentId` (where you came from), `keyElement`, `route` (web) / `null` (app), and `notes`. **Do this even if the feature is buggy** — the navigation map is the value later cases use to reach the feature. Regenerate `sitemap.json` at the end (`scripts/gen_sitemap.py`).
3. **Probe the main interactions** (oracle-first when a checklist exists, else heuristic): walk each spec requirement/business-rule/validation/UI-state and compare actual vs expected; plus the always-on heuristics — open/close, filter, create/edit, **open an item's detail**, validate **empty/boundary** input (negative values, max length, odd characters). Note what's correct/wrong, crashes, UI bugs, edge cases.
4. **Read figures/messages CRITICALLY**: wrong totals, nonsensical negative numbers, English text leaking through, raw SQL/exception/stacktrace exposed in the UI, context-wrong messages — **check computed figures against the spec's formulas/business rules** when the oracle gives them. Extract **verbatim** (via page source) to paste into the report.
5. **On-the-spot triage** ([failure-triage.md](../../rules/failure-triage.md)): each wrong behavior → reproduce manually on the app right away to decide **`[APP-BUG]`** (app is wrong vs spec/expectation — reproducible; tag *spec-mismatch* when it contradicts the oracle) vs `[FRAMEWORK]` (we grabbed the wrong element/automation) vs `[ENV]`/`[DATA]`. Behavior the spec doesn't cover → `[NEEDS-TRIAGE]` + question for PO. Only `[APP-BUG]` goes into the bug section.
6. **Capture evidence** → **always pass the FULL relative path as the screenshot tool's `filename`**, never a bare file name (a bare name lands in the MCP output dir / project root, not here). Web = `browser_take_screenshot` with `filename: "results/<feature-name>/screenshots/<BUG-ID>.png"`; app = `appium_take_screenshot` saved to the same path. **Named by BUG-ID** (e.g. `results/<feature-name>/screenshots/02-APP01-published-tab-crash.png`). Each `[APP-BUG]` ≥1 image; also save a few OK-screen images (`..-ok.png`) as proof of "checked".
7. **Bug report for the dev**: write `results/<feature-name>/dev-bug-report-<ddMMMyyyy>.md` in the **exact format** of [exploratory-bug-report-template.md](../../rules/exploratory-bug-report-template.md) — each bug: Screen · Symptom (verbatim) · Root cause (if found) · Impact · Expectation · Evidence · Defect ID. Include a **✅ Checked — NO bug** section + **❓ NEEDS-TRIAGE** + environment notes. Write the report in Vietnamese.
8. **Sync the register**: append each `[APP-BUG]` into the cross-feature register `results/bug-summary.md` (assign APP-ID, update the per-function count table + severity distribution).

## GATE decision
- 🔴 **Has `[APP-BUG]`** → the feature is NOT done. **Deliverable = bug report** for the dev. Do **NOT** proceed to `/qa:plan-tests`/`/qa:cook` for the broken part (only cleanly separable clean parts move on).
- 🟢 **No `[APP-BUG]`** (only possibly FW/ENV/DATA) → app is correct → extract clean elements → ready for **`/qa:plan-tests`**.

> *Automation runs only when the app is already correct.* Planning is the `plan-method` skill; automation errors (FW) are fixed via the `fix-by-layer` skill, NOT recorded in the app bug section.
