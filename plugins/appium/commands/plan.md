# /plan — Exploratory gate → Lập kế hoạch test

Người dùng yêu cầu: **$ARGUMENTS**

Mục tiêu: tạo **plan cứng (file .md)** mô tả cách triển khai test (Appium Java + TestNG + POM). Sau này `/cook` thực thi. **KHÔNG viết code** ở bước này.

> 🚦 **GATE bắt buộc — exploratory trước, plan sau**: *Automation chỉ hoạt động khi app đã đúng.* Vì vậy `/plan` **luôn chạy exploratory trước** (Bước 0). **Nếu tính năng có `[APP-BUG]` → KHÔNG lập plan test**; deliverable là **bug report** gửi dev. **Chỉ khi exploratory sạch (không `[APP-BUG]`)** mới lập plan test.

## Bước 0 — EXPLORATORY GATE (chạy trước mọi thứ)
1. Kiểm tra đã có bug report gần đây cho feature chưa: `reports/exploratory/<group>/dev-bug-report-*.md` + `reports/exploratory/bug-summary.md`.
   - Chưa có / đã cũ / feature đổi → **chạy `/exploratory <ten-tinh-nang>`** (mở app qua Appium MCP, săn bug) theo [exploratory.md](exploratory.md).
2. Đọc kết quả gate:
   - 🔴 **Có `[APP-BUG]`** cho feature (hoặc sub-feature) → **DỪNG lập plan** cho phần đó. Bảo đảm bug report đúng định dạng [exploratory-bug-report-template.md](../rules/exploratory-bug-report-template.md) + đã append `bug-summary.md`. In: "Feature CHƯA hoàn tất — N lỗi app, xem `dev-bug-report-*.md`, cần dev fix trước khi viết test." KHÔNG tạo `plans/<feature>/plan.md`.
   - 🟢 **Không `[APP-BUG]`** (chỉ có thể FW/ENV/DATA) → app đúng → **tiếp tục Bước 1+** để lập plan. Element đã trích ở exploratory dùng luôn cho Layer 1/2.
3. Feature lớn gồm nhiều sub-feature: **gate theo từng sub-feature** — phần sạch thì plan, phần lỗi thì báo bug; ghi rõ phần nào đi tiếp, phần nào chờ dev.

## Bước 1 — Đọc external docs (nếu có link Lark/Figma)
Nếu request chứa link Lark (`larksuite.com/wiki/`, `feishu.cn/wiki/`) hoặc Figma (`figma.com/design/`):
1. Tạo `plans/<feature>/` + `tracking.md` (header "Auto-generated by reader agents").
2. Spawn reader agent — **1 agent/URL**, song song, tối đa 5/batch:
   - Lark → agent `lark-reader` (`${CLAUDE_PLUGIN_ROOT}/agents/agent-lark-reader.md`).
   - Figma → agent `figma-reader` (`${CLAUDE_PLUGIN_ROOT}/agents/agent-figma-reader.md`, `max_screens: 7`).
   - Mỗi agent **append** kết quả vào cùng `tracking.md`.
3. Sau khi xong → đọc `tracking.md`; nếu Lark doc lộ link Figma mới chưa đọc → spawn thêm `figma-reader`. Chi tiết: [lark-mcp-guide.md](../rules/lark-mcp-guide.md).

## Bước 2 — Khảo sát & element lookup (3-layer)
- Tìm Screen/Test đã có của feature (`Glob src/**/<group>/**`) để tái dùng `base`/`utils`/`actions`/`models`, không trùng.
- Element lookup chain ([design-pattern §2](../rules/design-pattern.md)): **Screen.java** → **elements.json** → (nếu thiếu) ghi chú "cần MCP discovery" trong plan để `/cook` chạy agent `source-inspector`. KHÔNG mở device ở bước plan trừ khi cần.
- Mỗi Screen mô tả theo **skill `declare-screen`** (key element cho `isDisplayed()`, action = verb, no-assert). Element dự kiến thiếu `id` → ghi chú để `/cook` RECORD qua **skill `missing-ids`**.

## Bước 3 — Viết plan ra file `plans/<feature>/plan.md`
Folder lowercase-hyphen. Nếu đã có → cập nhật. Cấu trúc:
```markdown
## Objective
## Analysis  (file liên quan · tái dùng · file mới · nguồn element: Screen/JSON/MCP)
## Steps     (mỗi step: file path + create/edit + chi tiết)
## Screens   (| Screen | File | Key element (isDisplayed) | Locator chính | Action methods |)
## Test cases (| Test | Scenario | Assertion |)  — liệt kê FULL hàm test, mỗi hàm 1 scenario
## TestNG XML (GoToHomeTest là <test> block đầu)
## Missing ID Report  (| element | màn | locator hiện tại | mô tả | — bỏ nếu tất cả có id)
## Notes / rủi ro / câu hỏi mở
```

## Nguyên tắc
- Plan tuân thủ tách lớp: `screens` (no-assert) → `tests` (assert) → `regression/smoke` (compose, extends `BaseRegression`).
- **Feature bắt đầu từ Home**: mô tả luồng từ Home → vào tính năng. GoToHomeTest đầu TestNG XML (chỉ áp dụng lúc chạy test).
- Sitemap (skill `update-sitemap`): plan **liệt kê** màn/node cần tạo/cập nhật — `/cook` mới ghi thật.
- **GATE app-đúng (BẮT BUỘC)**: chỉ lập plan test cho feature/sub-feature **đã qua exploratory sạch** (Bước 0, không `[APP-BUG]`). Feature đang lỗi → **không có plan test**, chỉ có bug report gửi dev (automation chỉ chạy khi app đúng). Plan vẫn ghi **kỳ vọng đúng** (theo SBH Reactive/spec) cho mỗi test để `/cook` giữ test trung thực.
- Sau khi ghi file → in đường dẫn plan + tóm tắt 3-5 dòng để người dùng review trước khi `/cook`.
